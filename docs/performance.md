# Performance Guide

This document describes the performance tooling and runtime behavior for large
coverage datasets.

## Viewer large-table behavior

Point tables use two rendering modes:

1. Full mode for up to `50,000` rows.
2. Optimized large mode above `50,000` rows.

In large mode, expensive per-column features are reduced to keep UI interaction
responsive.

### Full-feature override

When a point table is above the large-table threshold, users can choose
**Enable full features**.

- Default: disabled.
- Guardrail: first use requires confirmation.
- Persistence: session-only (resets on reload/restart).

Use **Return to optimized mode** to switch back immediately.

## Python microbenchmarks

Run from repo root:

```bash
uv run python tools/perf/bench_axis_lookup.py --iterations 1000000
uv run python tools/perf/bench_bucket_hit.py --iterations 300000
```

These benchmark scripts cover:

- `Axis.get_named_value()` exact-value and range-heavy lookups.
- `Bucket.hit()` exact-axis and range-heavy coverpoints.

## Synthetic viewer fixtures

Generate a large fixture for manual viewer testing:

```bash
uv run python tools/perf/gen_large_viewer_fixture.py \
  --axis-a 100 --axis-b 100 --axis-c 20 \
  --output tools/perf/fixtures/large_200k.bktgz
```

The default dimensions above generate `200,000` buckets.

You can also emit JSON for direct inspection:

```bash
uv run python tools/perf/gen_large_viewer_fixture.py --output /tmp/large_200k.json
```

## Manual validation checklist

1. Load a large fixture and confirm optimized mode is used above `50,000` rows.
2. Confirm filtering/sorting in large mode remains responsive.
3. Enable full features and verify warning prompt appears once per session.
4. Return to optimized mode and confirm responsiveness recovers.
5. Verify behavior in both Electron and web runtime.
